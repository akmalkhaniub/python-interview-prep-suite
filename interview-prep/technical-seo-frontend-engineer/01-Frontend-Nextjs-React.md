# Frontend Development, Next.js & React

**Focus:** Next.js architecture, rendering strategies, building calculators, modern React

---

### Q1: Rendering Strategies in Next.js
**Question:** Explain the difference between SSR, SSG, ISR, and CSR in Next.js. Which would you use for a financial calculator page vs. a glossary of financial terms?

**Expected Answer:**
- **SSR (Server-Side Rendering)**: HTML is generated on the server on *every request*. Slower TTFB (Time to First Byte) but always fresh. Use when data changes constantly and SEO is still needed.
- **SSG (Static Site Generation)**: HTML is generated at *build time*. Extremely fast, cacheable on CDN. Best for SEO.
- **ISR (Incremental Static Regeneration)**: SSG but updates in the background after a specified timeout (`revalidate`).
- **CSR (Client-Side Rendering)**: HTML is a bare shell, JS fetches data and renders on the client. Bad for SEO.
- **Glossary Page**: Use **SSG** or **ISR** (if adding terms frequently). Content rarely changes, speed is paramount for SEO.
- **Calculator Page**: Use **SSG** for the outer shell/metadata (for SEO), but the actual calculator logic runs via **CSR** (React state) when the user interacts. Next.js App Router handles this elegantly with Server Components for the layout and Client Components for the calculator widget.

### Q2: Building Calculator Pages
**Question:** The JD specifically mentions building "calculator pages". How would you approach building a Compound Interest Calculator in React that is both performant and highly SEO-optimized?

**Expected Answer:**
- **SEO Shell**: The page must have a rich, text-based explanation of compound interest, formulas used, and FAQ schema rendered server-side so Google understands the page context.
- **Client Component (`"use client"` in Next.js 13+)**: The calculator itself needs to be interactive. 
- **State Management**: Keep it simple with `useState` or `useReducer` for inputs (Principal, Rate, Time, Contribution).
- **Performance**: 
  - Calculations should be optimized. If doing heavy math/charting, use `useMemo` so recalculations only happen when inputs change.
  - Dynamically import (`next/dynamic`) heavy charting libraries (like Recharts or Chart.js) so they don't block the initial page load or hurt Core Web Vitals.
- **UX/SEO overlap**: Update the URL parameters (e.g., `?principal=1000&rate=5`) as the user interacts. This allows users to share specific calculations, which drives backlinks and traffic.

### Q3: Next.js App Router vs Pages Router
**Question:** Have you worked with the Next.js App Router? How does it change how we build SEO-focused pages compared to the older Pages router?

**Expected Answer:**
- **React Server Components (RSC)**: By default, components in the App Router are Server Components. They send zero JavaScript to the client, reducing bundle size drastically—a massive win for Core Web Vitals.
- **Data Fetching**: We no longer use `getServerSideProps` or `getStaticProps`. We use standard `async/await` directly in the component, and Next.js patches `fetch` to handle caching and revalidation natively.
- **SEO Metadata**: Replaced `<Head>` with the Metadata API (`export const metadata = {}` or `generateMetadata()`). It's strictly typed and ensures metadata is streamed correctly in the `<head>` before the body renders.

### Q4: Reusable Component Architecture
**Question:** You need to build 50 different financial calculators over the next year. How do you architect this to avoid reinventing the wheel?

**Expected Answer:**
- **Design System / UI Kit**: Build a strict set of reusable input primitives: Currency Inputs, Slider Inputs, Percentage Inputs, and Result Displays.
- **Logic Extraction**: Extract common financial math formulas into pure, testable utility functions independent of React.
- **Layout Templates**: Create a standard `CalculatorLayout` component that takes `title`, `description`, `faqData`, and `children` (the specific calculator UI).
- **Headless Hooks**: Create custom hooks like `useAmortization()` that return data, keeping the UI completely separate from the math.

### Q5: Handling Hydration
**Question:** What is hydration in Next.js, and how do you prevent hydration mismatch errors when building dynamic pages?

**Expected Answer:**
- **Hydration**: The process where React attaches event listeners to the static HTML sent by the server, making it interactive.
- **Mismatch**: Happens when the HTML generated by the server differs from what React expects to render on the client's first pass (e.g., using `window.innerWidth` or `Date.now()` during initial render).
- **Fix**: Ensure the initial render is deterministic. If you need client-specific data (like a user's local timezone), render a skeleton or generic state first, then use a `useEffect` to set a flag (`isMounted`) to render the client-specific UI on the second pass.
